
## 2026-07-21 — 修复手工 URL 模式 Phase 2 不自动导航/过早生成 dump

### 用户反馈的问题
1. **Phase 2 没自动导航到 Phase 1 手工输入的 URL**（没“自动回车”）
2. **Phase 2 开启即生成 dump 文件**，应该等视频数据流开始后才生成

### 根因
- 当前 preheat 逻辑：http(s) URL 只有 60s 超时，超时后 fallback 到真 dump 路径
- 手工输入 URL 场景：用户可能没点播放、视频处于暂停态，60s 后强制进入主流程，dump 文件已创建但无数据流
- 没有区分“命令行已知 URL”（可短超时）vs “Phase 1 手工 URL”（需无限等待 playing）

### 修复内容

#### capture.sh
- Phase 1 结束时：等待 `sessionstore-backups/recovery.jsonlz4` 同步完成（最多 15s 轮询），提取真实 URL 写入 `$PROFILE/.phase1_url`
- Phase 2 启动时：识别 `.phase1_url` 存在 → 设置 `MANUAL_URL_MODE=1` 传给 `monitor_run`
- Phase 2 日志显示“手工 URL 模式：等待用户点击播放后再开始抓取”

#### lib-monitor.sh
- 新增 `_monitor_wait_real_video_manual()`：无限轮询 `state=playing`，无超时 fallback
- `monitor_run` 接收第 7 参数 `manual_url_mode`
- 手工模式：`preheat_timeout=0` → 调用 `_monitor_wait_real_video_manual()`，检测到 playing 后 kill+restart Firefox 用真 dump 路径
- 普通 http(s) URL 预热也要求 `state=playing`（避免暂停态误判 ready）

### 核心流程（手工 URL 模式）
```
Phase 1: 用户手工输入 URL + 过 CF → SIGTERM 保存 session
         ↓ 等 sessionstore.jsonlz4 同步（≤15s）
         提取真实 URL → 写 $PROFILE/.phase1_url

Phase 2: 读 .phase1_url → MANUAL_URL_MODE=1
         启动 Firefox (dump=/dev/null) + BiDi daemon
         ↓ 无限等待 state=playing（用户点播放）
         检测到 playing → kill Firefox + 重启 (dump=真路径) + seek 当前时间
         → 进入主监控循环（playing/ended/stall...）
```

### 关键技术点
- **StreamDumper 是 firefox 进程内 static 状态**：首次触发写 SPS/PPS 后整个生命周期不再重写
- 预热阶段 `MOZ_STREAM_DUMP_PATH=/dev/null` 避免 about:home/中间页/广告污染 dump
- 真视频 playing 后 kill+restart = StreamDumper 干净初始化，写真视频 SPS/PPS
- `manual_url_mode=1` 时无超时 fallback，绝不生成空 dump 文件
