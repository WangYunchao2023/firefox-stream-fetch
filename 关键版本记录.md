## v3.1.0 (2026-07-17) — 当前版本

**自动连集：每集独立 mp4、URL+src 双信号切集检测、顺序命名**

| 维度 | 内容 |
|---|---|
| **核心成果** | `--auto-next` 正式可用：playlist 自动连集，每集独立 `host-01.mp4`, `host-02.mp4`... |
| **切集检测** | BiDi 轮询时双信号确认：`location.href` 变化 + `<video>.currentSrc` 变化 + `readyState >= 2` |
| **数据流** | monitor 写 sidecar `next_url` + `episode_index` → capture.sh 循环读取 → Phase 2 重跑 |
| **命名** | 固定顺序命名 `{host}-{02d}.mp4`（无需额外参数） |
| **回退** | 无下一集 / 用户未开 `--auto-next` → 单集正常结束 |

## v3.0.0 (2026-07-17)

**核心成果**：统一抓取入口，保留两阶段架构，环境变量代理，兼容旧脚本。

| 维度 | 内容 |
|---|---|
| **核心成果** | 4 个 site 适配脚本合并为 1 个 `capture.sh`；旧脚本保留为 1 行 shim 兼容
| **架构** | Phase 1（过验证/加载，sandbox 开） → Phase 2（抓流，sandbox 关）两阶段保留
| **代理策略** | 统一读 `http_proxy/https_proxy/all_proxy` 环境变量；Phase 1 显式 unset 避免 CF 检测
| **输出目录** | 默认 `~/Videos/firefox抓取/`，文件名 `<host>-<timestamp>.mp4`
| **监控** | lib-monitor.sh 智能监控（BiDi 5s 轮询 video.state + ended/stall/paused/no-video + 自动恢复 ≤3 次）
| **反指纹** | Phase 1 最小 prefs（真实 UA + sandbox 开），Phase 2 关 sandbox 允许写盘
| **音视频** | H.264/HEVC/AV1 自动 dispatch + AAC 双轨，合流 ffmpeg `-c copy`
| **兼容** | `capture-olevod.sh/youtube.sh/yfsp.sh` 透传参数到 `capture.sh`
| **验证** | olevod.com 明文流端到端跑通（178MB h264 + 11MB aac → mp4）

## v2.2.0 (2026-07-16)

**核心成果**：Phase 1 → Phase 2 真正能自动切换。

| 维度 | 内容 |
|---|---|
| **解决** | capture-generic.sh Phase 1 启动时带了 `-remote-debugging-port` 和 BiDi 检测，CF Turnstile 会检测 devtools-protocol 端口作为 bot 信号，导致 yfsp.tv 等网站反复弹人机验证；biDi 轮询查 video state 也不可靠 |
| **效果** | Phase 1 走纯 X11 窗口标题检测（与稳定的 capture-yfsp.sh 一致），CF 过了、页面跳转、窗口标题变 → 自动 SIGTERM firefox 保存 session → Phase 2 接上。同 profile + cookie 复用，dump 产双轨 mp4 |
| **关键改动** | `scripts/capture-generic.sh` Phase 1 去掉 `-remote-debugging-port` 和 BiDi 检测，改用 xdotool 窗口标题（“Just a moment...” → 视频标题）；`get_firefox_wid` 优先匹配 "Nightly" 标签页窗口；`bidi-state.py` 补充 `no-video` 状态下的 `url` 字段 |
| **端到端验证** | yfsp.tv `https://www.yfsp.tv/watch?v=ztgsSWh5mPZEhhazLjYUG6` 实操：18s 过 CF → Phase 1 → Phase 2 自动切换 → h264 19.8MB + aac 415KB → mux 成 320x240 H.264 + AAC mp4 (2:21, 20MB) → 抽帧确认是 "Linya bilibili" vlog 真实内容 |

## v2.1.0 (2026-07-15)

**核心成果**：音视频双轨抓取，真正可用。

| 维度 | 内容 |
|---|---|
| **解决** | 之前 dump 只有画面没声音，抓下来的 mp4 静音 |
| **效果** | 跟 H.264/HEVC/AV1 同一个 hook 点同时 dump AAC，ffmpeg `-c copy` 合流即得完整 mp4 |
| **关键改动** | `patches/0004-add-audio-dump.patch`：`DumpAudio()` 复用 `HandleDemuxedSamples` hook，按 TrackType 分支；路径 trim 已知视频扩展名输出 `<base>.aac` |
| **端到端验证** | HTTP 加载测试流 → `.h264 + .aac` 双 dump → 合流 mp4 双轨 → astats Peak -18dB / RMS -22dB / FFT 主频 440Hz（与源完全一致）|
| **附带 fix（未发 tag）** | `e852ad9` — capture-generic.sh 加 `MOZ_ENABLE_WAYLAND=0` + `GDK_BACKEND=x11`，修 GDM X11 session 下窗口不显示的 race condition |

## v2.0.0 (2026-07-14)

**核心成果**：CF Turnstile 反反爬，通用化抓取脚本。

| 维度 | 内容 |
|---|---|
| **解决** | Cloudflare Turnstile 检测 prefs 特征（Chrome UA / 全 sandbox 关闭）→ 不发 cf_clearance cookie，自动化抓不到内容 |
| **效果** | "两阶段法"：Phase 1 sandbox 默认 + 真实 UA 让用户手动过 CF → cookie 落盘；Phase 2 同 profile sandbox 关闭 + cookie 复用 → 自动化抓取。同一 cookie 复用可持续几小时 |
| **关键改动** | `scripts/capture-generic.sh`：通用 CF 抓取脚本，URL 可选 / 多集自动 / session restore；`scripts/capture-yfsp.sh`：yfsp.tv 专用（WebGL 稳定 prefs + 代理修正）|
| **工程改进** | find 不跟 symlink；启动后拉窗口到前台；WebDriver BiDi 兼容；输出目录配置化 |
| **实战验证** | 7/14 晚用 yfsp 抓到 14 分钟 H.264 视频（`yfsptv-20260714-210831.mp4`，30MB）|

## v1.1.0 (2026-07-13)

**核心成果**：从 H.264 单 codec 扩展到多 codec，加 site 适配脚本。

| 维度 | 内容 |
|---|---|
| **解决** | YouTube 等网站默认走 AV1，原版只支持 H.264 → 抓到 0 字节 |
| **效果** | 自动按 codec dispatch，AV1 自动包 IVF 容器，ffprobe/ffplay 直接认；HEVC 走 Annex B |
| **关键改动** | `patches/0002-add-av1-support.patch` + `patches/0003-codec-aware-output.patch`：codec mime 探测 + IVF 32-byte 文件头 + 12-byte frame header 包装 |
| **附带产物** | `scripts/capture-{youtube,olevod,yfsp}.sh` 三个 site 适配脚本（YouTube AV1 / olevod 直连 / yfsp CF）|
| **实战验证** | YouTube `_n4SRDYkhqs` 抓到 39MB IVF/AV1 真实内容（ffprobe 抽帧确认是两人在工作室）|

## v1.0.0 (2026-07-12)

**核心成果**：项目从 0 到 1，端到端可用。

| 维度 | 内容 |
|---|---|
| **解决** | 浏览器 DRM 解密后的原始视频码流无法获取 → 想抓视频必须重新录屏（firefox-re 项目方案，有损） |
| **效果** | 在 CDM 解密后 / 解码前 hook 拿到 H.264 Annex B 裸流，ffplay 直接播、ffmpeg `-c copy` 转 mp4，**画质与原网站等同（无损）** |
| **关键 hook 点** | `MediaFormatReader::HandleDemuxedSamples`（MP4Demuxer 路径）+ `OnVideoDemuxCompleted`（MSE/EME 路径）双 hook |
| **关键工程坑** | MP4Demuxer 不设 `sample->mTrackInfo` → hook 必须传 `decoder.GetCurrentInfo()` 兜底；TrackType 完全限定名是 `mozilla::TrackInfo::TrackType`（内嵌 enum）；mozconfig 用 `MOZ_OBJDIR` 绝对路径避免 firefox 源内创建 obj 目录 |
| **实战验证** | olevod.com `https://www.olevod.com/player/vod/1-82695-1.html` 抓到 18MB / 89s / 1280x720 H.264 High Profile，ffprobe + 抽帧确认内容真实 |

## 当前已知限制（待后续版本解决）

1. **Phase 1 退出检测只靠 tab 标题变化** → 通用性差，建议改成 cookie + video.readyState 多信号
2. **Phase 2 完成检测是"N 秒无 dump 增长"** → 网速慢会误判，建议改成 video.ended + buffered.end 双重检测
3. **只能 L3 软件解密**（桌面 Linux 默认）→ L1 hardware 解密拿不到
4. **无 EME+Widevine 真实内容验证**（之前 olevod 是明文流）→ 待真实 Widevine 内容验证
